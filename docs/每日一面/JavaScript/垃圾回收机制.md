# JavaScript 进阶 - 垃圾回收机制

## 简介

JavaScript 是在创建变量（对象、字符串等）时自动进行内存分配，并且在不使用它们时“自动”释放。释放的过程称为垃圾回收。

## 内存生命周期

1. 分配你所需要的内容
2. 使用分配到的内容（读、写）
3. 不需要时，释放内存

```js
const n = 123; // 为数值分配内存
const s = "azerty"; // 为字符串分配内存

const o = {
  a: 1,
  b: null,
}; // 为对象及其包含的值分配内存

// 为数组及其包含的值分配内存（就像对象一样）
const a = [1, null, "abra"];

function f(a) {
  return a + 2;
} // 为函数（可调用的对象）分配内存
```

## 引用

垃圾回收算法依赖的主要概念是引用。在内存管理的上下文中，如果一个对象有访问另一个对象的权限（隐式或者显式），称作前面的对象引用后面的对象。例如，一个 JavaScript 对象具有对它的原型的引用（隐式引用）和对它属性的引用（显式引用）。

## 引用计数垃圾回收

引用计算垃圾回收是最初级的垃圾回收算法，此算法把“对象是否不再需要”简化定义为“对象有没有其他对象引用到它”。如果没有引用指向该对象（零引用），对象将被垃圾回收机制回收。

当一个对象被引用时，它的引用计数会增加；当一个对象不再被引用时，它的引用计数会减少。当引用计数为 0 时，表示该对象不再被引用，JavaScript 引擎会将其回收。

```js
let x = {
  a: {
    b: 2,
  },
};
// 创建了两个对象。一个作为另一个的属性被引用。
// 另一个被赋值给变量‘x’。
// 很显然，没有可以被垃圾回收的对象。
let y = x;
// 变量‘y’是第二个拥有对象引用的变量。
x = 1;
// 现在，起初在‘x’中的对象有唯一的引用，就是变量‘y’。
let z = y.a;
// 引用对象的‘a’属性。
// 现在，这个对象有两个引用，一个作为属性，
// 另一个作为变量‘z’。
y = "mozilla";
// 起初在‘x’中的对象现在是零引用了。它可以被垃圾回收了。
// 但是，它的属性‘a’仍被变量‘z’引用，所以这个对象还不能回收。
z = null;
// 起初在 x 中的对象的属性‘a’是零引用了。这个对象可以被垃圾回收了。
```

循环引用是引用计数垃圾回收算法的一个问题，是导致内存泄漏的常见原因。

```js
// 创建了两个对象，一个对象的属性引用另一个对象，形成了一个循环
function f() {
  const x = {};
  const y = {};
  x.a = y; // x 引用 y
  y.a = x; // y 引用 x
  return;
}

f();
```

## 标记清除算法

标记清除算法是现代浏览器中常用的垃圾回收算法。这个算法把“对象是否不再需要”简化定义为“对象是否可以获得”。

从根对象（window全局）开始查找所有引用的对象，并标记为“使用中”，没有标记为使用中的对象就是垃圾对象。

现代 JavaScript 引擎（如 V8、SpiderMonkey）通常使用标记-清除算法进行垃圾回收。它通过从根对象开始遍历所有对象，标记所有可以访问到的对象，然后清除未标记的对象。这样，所有无法被访问到的对象都会被回收。

这种算法很好解决了循环引用的问题，循环引用的对象也可能是垃圾对象。不过这种算法会导致那些无法从根对象查询到的对象都将被清除。


## 内存泄漏

内存泄漏是指程序中已经不再需要使用的内存，却没有被释放的情况。在 JavaScript 中，常见的内存泄漏包括未被清除的定时器、未被移除的事件监听器、闭包中的变量等。

# 2026/07/01 更新垃圾回收机制

之前对垃圾回收机制的理解不够好，这里更新下。

## 简介

JavaScript 的垃圾回收是一种自动的内存管理机制。它的核心目的是找出那些不在被程序使用的变量和对象，并释放它们所占用的内存空间，以防止内存泄漏、 保证程序的运行性能。

## 核心算法

从“引用计数”到“标记清除”，再到现代的“分代回收”。

### 引用计数

跟踪记录每个值被引用的次数。次数为0时就回收。当一个对象被引用时，它的引用计数会增加；当一个对象不再被引用时，它的引用计数会减少。当引用计数为 0 时，表示该对象不再被引用，JavaScript 引擎会将其回收。

引用计数存在一个致命缺点：循环引用。如果对象 A 引用了 B，B 也引用了 A，两者的引用计数永远不为 0，导致内存泄漏。

### 标记清除法（可达性分析）

从根对象（window全局）开始查找所有引用的对象，并标记为“使用中”，没有标记为使用中的对象就是垃圾对象。

它通过从根对象开始遍历所有对象，标记所有可以访问到的对象，然后清除未标记的对象。这样，所有无法被访问到的对象都会被回收。

### 分代回收

V8 引擎基于“代际假说”，认为大部分对象在内存中存活的时间都很短（比如函数执行完，内部变量就死了）。

因此，V8 引擎将内存分为新生代和老生代两个区域，分别采用不同的垃圾回收算法。

新生代：存放存活时间短的对象（如局部变量）。

采用 Scavenge 算法（即复制算法），将内存一分为二（From 空间和 To 空间）。每次把 From 空间里的存活对象复制到 To 空间，然后清空 From 并对调角色。这种空间换时间的做法速度极快。

老生代：存放存活时间长或常驻的对象（如全局变量，闭包变量）

采用 标记清除（Mark-Sweep） 和 标记整理（Mark-Compact）。因为老生代对象多，为了防止内存碎片化，需要把存活的对象往内存的一端移动，然后清理掉边界以外的内存。

## V8 引擎的优化

“Stop-The-World”（全停顿）是 GC 无法避免的副作用。在执行垃圾回收时，JS 线程必须暂停主线程的执行，这可能导致页面卡顿。因此，V8 引入了以下优化机制：

- 增量标记：把一次完整的、耗时长的标记过程拆分成很多小步，穿插在 JS 主线程逻辑之间执行，避免长时间的视觉卡顿

- 并发/并行回收：利用辅助线程在后台进行标记和清理，充分利用多核 CPU

## 内存泄漏

内存泄漏是指程序中已经不再需要使用的内存，却没有被释放的情况。在 JavaScript 中，常见的内存泄漏包括未被清除的定时器、未被移除的事件监听器、闭包中的变量等。
实际开发中，应该注意以下几个方面：

- 及时解除引用：对于全局变量、长生命周期的对象，不需要时手动赋值为 null

- 善用弱引用

- 注意闭包和定时器

## 总结

总的来说，JS 的垃圾回收是以标记清除算法（可达性分析）为基础，在 V8 引擎中通过分代回收来实现高效运转。

## 一句话讲清

简单来讲，JavaScript 的垃圾回收就是一种自动运行的内存管理机制，它的核心目的很简单：找出那些不再被使用的变量和对象，把它们占用的内存释放掉，防止内存泄漏。

在实现层面上，以前的老浏览器用过一种叫‘引用计数’的方法，就是看一个对象被引用了多少次。但它有一个致命的缺点，就是处理不了循环引用，容易导致内存泄漏，所以现代引擎基本不用了。

现在的现代浏览器（比如 Chrome 的 V8 引擎），核心使用的是‘标记清除算法’，它基于一个概念叫‘可达性分析’。

怎么理解呢？引擎会定期从一个‘根对象’（也就是全局的 window 或者 global）开始向下遍历。只要能从根节点访问到的对象，就叫‘可达的’，说明还在用，留下来；如果一个对象无论如何都访问不到了，哪怕它们之间自己互相引用，也会被判定为‘不可达’的垃圾，在随后的阶段被直接清理掉。

而且，V8 引擎在底层还做了一个非常聪明的优化，叫作‘分代回收‘。它把内存分成了新生代和老生代。

新生代里面放的都是些生命周期短的变量，比如函数执行完就没用的局部变量，V8 采用复制算法将内存一分为二（From 空间和 To 空间）。每次把 From 空间里的存活对象复制到 To 空间，然后清空 From 并对调角色。这种空间换时间的做法速度极快。

而那些经历了几次回收还存在的对象，或者全局变量，就会被晋升到老生代。老生代里会用‘标记清除’和‘标记整理’，甚至会利用多线程、增量标记等手段，来防止垃圾回收时导致页面卡顿。

在实际开发中，我们需要注意几个点：第一，是在组件销毁或者不需要的时候，及时清理定时器、解绑全局事件；第二，就是善用 WeakMap 和 WeakSet。因为它们对 Key 是弱引用。
