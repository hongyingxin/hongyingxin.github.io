## 6月17日

### jenkins 配置

在配置 Jenkins 的过程中，卡在`pnpm install`上，把依赖更新之后，还是不行。报错信息如下：

```bash
Cannot install with "frozen-lockfile" because pnpm-lock.yaml is not up to date with package.json
```

这才发现，没有把`pnpm-lock.yaml`文件也上传到仓库。

`pnpm-lock.yaml`文件是`pnpm`的锁定文件，记录了项目依赖的版本信息。为了保证生产环境的稳定，需要将`pnpm-lock.yaml`文件也上传到仓库，不能`.gitignore`忽略掉。

### pnpm11 各种问题

pnpm 11 在底层重构和供应链安全方面进行了重大升级（如使用 SQLite 替代旧存储，默认限制生命周期脚本执行），但这也导致了许多兼容性问题和破坏性变化。

1. 运行/升级时报错，提示 Node.js 版本过低

- 问题： pnpm 11 强制要求 Node.js ≥ 22。如果你的本地环境或 CI/CD 流水线还在使用 Node 18 或 20，安装和运行会直接报错。

- 解决： 将 Node.js 升级至 22+。如果是 CI/CD 运行报错，需修改构建环境配置（如 GitHub Actions 中的 node-version: 22）。

2. 过去写在 .npmrc 中的配置失效

- 问题： pnpm 11 不再从 .npmrc 中读取非认证类配置。以前用于控制依赖提升（如 shamefully-hoist）的设置可能会被全部忽略。

- 解决： 将自定义配置（如提升策略等）移至项目根目录的 pnpm-workspace.yaml 或全局的 ~/.config/pnpm/config.yaml 中。.npmrc 现在仅保留认证信息和注册表源（registry）。

3. 安装包含 postinstall 的包卡住或报错

- 问题： pnpm 11 默认收紧了安全设置，默认阻止依赖项的生命周期脚本（如 install、postinstall，常见于 sharp、esbuild 等涉及原生编译的包），需要明确授权。

- 解决：执行 pnpm approve-builds 进行交互式授权。或在 pnpm-workspace.yaml 中显式配置允许构建的包：

```yaml
allowBuilds:
  - sharp
  - esbuild
```

4. 新发布的包无法被安装（提示“包太年轻”）

问题： pnpm 11 为了防范供应链投毒，引入了发布“冷静期”机制（minimumReleaseAge，默认新包需发布一段时间后才允许被解析安装）。

解决： 如果急需安装刚发布的热修复版本，可在 pnpm-workspace.yaml 中关闭此限制：

```yaml
auditConfig:
  minimumReleaseAge: 0
```

5. onlyBuiltDependencies 报错

- 问题： 以前用于管理生命周期脚本的 onlyBuiltDependencies 在 v11 中被废弃。

- 解决： 使用新的 allowBuilds 规则进行替代（见问题 3 的配置方式）。

6. 全局安装（pnpm add -g）的相关问题

- 问题： pnpm 11 不再支持全局包共用依赖（彻底隔离），且全局二进制文件存放路径改变。升级后老版本的全局二进制可能会失效。

- 解决： 运行 pnpm setup 重新配置系统环境变量 PATH，随后需要通过 `pnpm add -g <pkg>` 逐个重新安装以前全局安装过的依赖包。

7. Windows 系统更新失效

- 问题： 在 Windows 上使用 pnpm self-update 升级时可能会出现路径 bootstrap 加载器未更新的 Bug（依然指向旧版本）。

- 解决： 建议使用 npm 重新安装或通过 Corepack 来安装和更新 Windows 端的 pnpm。
