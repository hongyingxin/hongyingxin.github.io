## 5月6日

今日的问题是：Vue和React的区别是什么

这里我们可以理解为汽车的全自动挡和手动挡的区别。

**设计哲学**

Vue是渐进式框架，插件主要由官方维护，主张关注点分离，通过`v-model`、`v-html`等指令降低了开发心智负担

React是函数式编程，一切逻辑皆函数，只负责视图层，生态广泛但碎片化，社区插件多，需要对JS有好的理解

**响应式与更新机制**

Vue基于`Proxy`通过依赖追踪和派发更新实现响应式，数据变化自动更新视图，不需要手动优化渲染

React不可变性和调和，数据状态变化重新执行渲染函数，引入`Fiber`架构切片，需要开发去优化

**编译优化**

Vue在编译阶段做了大量优化，通过静态提升、补丁标记等方式，减少了运行时的Diff压力

React不做编译优化，重点在于运行时的并发调度

**总结**

在AI Coding的今天，哪种框架好并不重要，而重要的是编程思维。一个零基础的Vue开发者，借助AI可以快速转换到React，反之亦然。

Vue的优势在于开发效率和一致性。在团队水平参次不齐的情况下或者需要快速交付、中后台系统等场景下，Vue凭借其双向绑定的指令和开箱即用等优势能够避免很多低级性能错误

React的优势在于逻辑的极致复用和上限。在做复杂的交互如在线文档、低代码平台等场景下，React的函数式编程和Fiber的调度能力，能够在架构设计上拥有更好的自由度。

## 5月12日

今天补充下性能优化的知识。

### defer和async的区别。

在默认情况下，浏览器遇到`<script>`标签会暂停解析HTML，下载并执行脚本，再继续解析HTML。这时候如果脚本执行时间过长，则会导致页面白屏时间过长。

async（异步）脚本下载时不阻塞HTML解析，但一旦下载完成，立即执行，执行脚本时会暂停HTML解析，不能确保执行顺序。常用于广告、埋点统计、第三方工具等场景。

defer（推迟）脚本下载时也不阻塞HTML解析，直到HTML完全解析完后再执行，执行顺序确定，常用于有依赖关系的脚本场景，如`jQuery`、`lodash`等。

async和defer是针对白屏时间的优化。

### 性能优化的另一个角度

前端性能优化是一个庞大的系统工程，在面试过程中可能很难全面回答，或者是遗漏掉一些重要的优化点。

今天换个维度来对性能优化进行总结。从**构建时**和**运行时**两个角度来对性能优化进行总结。

构建时即编译阶段，包括：

- 压缩混淆：可以引入`terser`插件进行极致的压缩，剔除`console`、`debugger`等无用代码

- `Tree Shaking`：得益于ES模块的特性，移除未使用的导出，减小最终bundle大小

- 代码分割：Vite默认开启，也可以配置`chunks`模块拆分

- 提取公共依赖：`Vue`、`Vue-Router`等第三方库可以使用`vite-plugin-cdn-import`插件提取到CDN外部引入，减少bundle大小

- 图片压缩：使用`imagemin`插件对图片进行压缩

- 开启Gzip压缩：使用`compression`插件对打包文件进行gzip压缩

- 缓存构建产物：使用`cache`插件缓存构建产物，减少构建时间

运行时即执行阶段，包括：

- CDN引入：将第三方库引入到CDN外部，减少服务器压力，提高加载速度

- 缓存策略：这里的缓存策略主要指HTTP请求缓存，包括强缓存和协商缓存

- 图片懒加载：使用`lazyload`插件对图片进行懒加载

- 字体优化：使用`font-display: swap`避免字体加载阻塞文本渲染

- 骨架屏：引入骨架屏，减少页面白屏时间

- 重绘重排：减少不必要的重绘重排，合并DOM更新，避免layout thrashing

- 防抖节流：限制高频操作执行频率，如搜索框输入、滚动事件等

- 虚拟列表：对于大量数据的列表，使用虚拟列表，只渲染可视区域的内容，减少DOM操作，提高性能

- Web Worker：将耗时计算完全移出主线程，使用Web Worker处理大型数据集、执行复杂计算或解析大量数据

以上，希望在面试过程提供另一个角度的性能优化思路。

## 5月14日

今天遇到个有趣的点。

用`v-html`动态插入`HTML`元素时，添加的类名样式不会生效。

这是因为在 Vue 的 scoped 样式中，通过 `v-html` 动态插入的 HTML 元素不会被加上 `data-v-xxxx` 属性，导致普通的 CSS 选择器无法匹配到它们。

所以需要使用深度选择器来穿透 scoped 样式，`:deep(.className)`。
