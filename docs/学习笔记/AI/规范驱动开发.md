# 规范驱动开发

## Prompt 提示词工程

Prompt 指提示词工程。

在提示词开发时代，如果直接对 AI 说：“开发一个 XXX 功能”，AI 大概率能立即完成。但只要继续往下开发，就会遇到几个明显的 AI 开发瓶颈。

## AI 开发瓶颈

### 1. 代码腐化和精神分裂

多次对话后，AI 会忘记之前的约定，一会儿用 A 库，一会儿用 B 库，代码风格越来越乱。

### 2. 上下文过长和 AI 幻觉

项目稍微大一点，把整个工程喂给 AI 会消耗巨量 Token，而且容易出现 AI 幻觉。

### 3. 改 A 塌 B

AI 缺乏全局架构观，为了修一个局部 Bug，可能导致另一个 Bug。

## SDD 规范驱动开发

SDD（Spec Driven Development）即规范驱动开发。

一个标准的 SDD 流程，通常会把开发拆分为四个严密阶段，把自然语言逐步具象化为代码：

```text
[人类想法] -> 1. Proposal（做什么） -> 2. Design（怎么做） -> 3. Tasks（怎么改） -> 4. Coding（AI 执行）
```

### 1. Proposal：提议阶段

提议阶段用于共同明确“要做什么”。

### 2. Design：设计阶段

设计阶段先让 AI 根据现有架构，产出一份详细的技术方案，用来对齐实现颗粒度。

### 3. Tasks：任务拆解阶段

任务拆解阶段把设计方案变成一个原子化的、可勾选的 TODO 列表。

### 4. Coding：编码实施阶段

编码实施阶段由 AI 根据任务列表进行实现和交付。

## Spec 工作流框架

### OpenSpec

OpenSpec 是目前开源社区比较热的轻量级 SDD 框架。

它的核心哲学是“流动的，而非死板的”。

OpenSpec 会把每次变更变成一个独立的文件夹，例如：

```text
openspec/changes/name
```

这个文件夹中通常包含：

- `proposal.md`
- `design.md`
- `tasks.md`

这种方式可以实现变更隔离，让每次需求、设计和任务都围绕同一个变更进行组织。

### Spec Kit

Spec Kit 是 GitHub 官方主导的开源 SDD 工具包，整体流程更具工业规范和严密性。

它的哲学是：“规范不服务于代码，代码服务于规范”。

工作流程通常从 `/speckit.constitution` 开始，用于确立指导原则。之后通过规范、规划、任务生成和实现进行推进，严格遵循：

```text
Specify -> Plan -> Tasks -> Implement
```

### GSD

GSD 将自己定位为执行优先的上下文工程系统。

它的哲学是优先考虑交付结果，而不是流程开销。

GSD 的特点是多 Agent，每个执行单元都有自己的上下文窗口。因此它更偏向于 Agent 级执行工具。

## 基于文件系统的隐式规范

### `.cursorrules`

`.cursorrules` 是cursor编辑器内置的一种规则，用来约束AI的行为。

### Claude Task Master / Taskmaster AI

Claude Task Master、Taskmaster AI 这类工具，很多能力已经被 Cursor 内置工作流替代。
