# 一篇文章彻底搞定性能优化

性能优化是前端面试过程中最高频的题目之一，但因为优化点太多过于零散，而我们只能靠背书式回答。

所以本文将原本零散的十几个优化点，归纳成一条主线，两大核心指标，三个空间维度，一篇文章彻底搞定性能优化。

## 总纲

面试官问：“请谈谈你做过哪些白屏和首屏的优化？”

面试官问：“谈下你做过的性能优化”

回答：

对于性能优化，我通常会围绕 FP（白屏） 和 LCP（首屏） 这两个核心指标展开。在实际项目中（比如 Ditto 项目），我会把优化拆分为三个空间维度：『构建时（编译阶段）』、『网络传输』、以及『运行时（执行阶段）』。

我们的核心逻辑是：在构建时算计好最轻量的产物，通过高效的网络管道分发，最终在运行时以最高优先级呈现。” 

> FP：白屏时间 FP 是指用户输入地址到浏览器屏幕上出现第一个像素的时间。通常认为浏览器开始渲染`<body>`标签或者解析完`<head>`标签作为白屏结束的标志。

> LCP：首屏时间 LCP 是指浏览器从响应用户输入网址，到首屏内容渲染完成的时间。

### 第一步：构建时优化

首先是构建时（编译阶段），这是优化的起点，核心是减少包体体积。

我利用了 `rollup-plugin-visualizer` 分析依赖，针对第三方组件库利用 `unplugin-vue-components` 实现按需引入。虽然 Vite 默认支持基础的代码分割和压缩混淆，但为了配合长效缓存，我手动配置了 `manualChunks` 进行精细化分包，把稳定的核心库抽离。同时，在配置中切换`ESbuild`到定制能力更强的 Terser，以便在生产环境完全剥离 `console.log` 和 `debugger`。

针对图片这个大头，我们在开发阶段使用 TinyPNG 针对性地手动压缩核心 UI 资产；在编译流水线中集成 `vite-plugin-imagemin` 插件进行自动化兜底压缩，确保没有漏网的大图。”

最终，配合 `vite-plugin-compression` 直接生成 Brotli/Gzip 高密度预压缩包，并使用 `vite-plugin-cdn-import` 将大库抽离为 CDN 引入。我们在构建时准备好的一切，都是为了给接下来的网络传输减轻压力。”  

**对应的优化点：**

- 使用 `rollup-plugin-visualizer` 分析依赖

- 使用 `vite-plugin-compression` 进行Gzip压缩（brotli进一步）

- 使用 `unplugin-vue-components` 对第三方组件按需引入

- 使用 `vite-plugin-cdn-import` 引入CDN

- 使用 `vite-plugin-imagemin` 对图片进行压缩

- 用 TinyPNG 手动压缩

- 压缩混淆，Vite默认开启，使用 Esbuild，Terser的定制能力更强，剥离`console.log`和`debugger`

- Tree Shaking优化，基于`import/export`，默认支持

- 代码分割，基于路由/异步引入，默认支持，可以使用`manualChunks`进行手动分割

### 第二步：网络传输优化

有了瘦身的产物后，第二维度的重点是网络传输，加快资源响应速度。

在 Ditto 项目中，静态资源全部托管到 CDN 加速，全量推行 HTTP/2 协议，利用其多路复用和头部压缩特性，解决浏览器的并发连接限制。

并且使用 HTTP 缓存策略，采用强缓存和协商缓存相结合的方式。

同时，我写了一个 Vite 插件，在编译时自动为 HTML 内的 CDN 资源注入 `<link rel="dns-prefetch">` 和 `<script defer>` 属性，提前解析 DNS。 

在我的另一个项目里，我更激进地落地了 Service Worker（依托 Workbox 生态）。它直接常驻在浏览器的网络层拦截 fetch 请求，通过预缓存（Precaching）清单实现本地 Cache Storage 的 0ms 离线响应，在网络层把白屏时间（FP）逼近了物理极限。”  

**对应的优化点：**

- HTTP缓存 强缓存 协商缓存

- CDN加速分发

- HTTP2.0 多路复用 头部压缩

- Service Worker（另一个项目使用）

### 第三步：运行时优化

最后，在运行时（加载与执行阶段），我们的核心策略是打破默认阻塞，抢占资源优先级

为了让浏览器尽快解析完 `<head>` 开始渲染 ，我们设计了一套动态资源预加载机制。比如针对首屏的 Banner 头图，我们通过脚本在 `requestAnimationFrame` 中动态创建 `<link rel="preload">` 标签，并 prepend 插入到 head 顶部。这样既提升了关键图片的解析优先级，又避免了 JS 脚本执行阻塞主线程。

在榜单列表页，我们使用了图片动态裁切，阿里云/七牛云OSS镜像，拼接参数裁剪，将几兆的原图在边缘侧动态降级为几十 KB 的 缩略图，并判断浏览器是否支持WebP格式，如果支持，则使用Webp格式。

我在另一个项目，除了常规的骨架屏、字体层面的 font-display: swap 外，在处理复杂计算的项目中，我落地了 Web Worker 多线程计算。把繁重的数据解析和逻辑计算任务通过 postMessage 移出主线程，让主线程全速处理 UI 渲染。

当然，常规的运行时基本功我们也做得非常扎实，比如全局的图片懒加载、针对高频事件的防抖节流、减少不必要的 DOM 重绘重排、防范长列表的虚拟列表。

Vue 框架层面的使用`v-key`、`v-once`与`v-memo`结合优化长列表、`v-if`和`v-show`结合懒加载、`keep-alive`缓存组件等。

**对应的优化点：**

- 针对首屏 Banner 头图，动态资源预加载，`requestAnimationFrame` `<preload>`

- 写了一个vite插件，对CDN资源加`<defer>`属性和`<dns-prefetch>`属性

- 图片动态裁切 阿里云/七牛云OSS镜像，拼接参数裁剪

- 重绘重排（非重点）

- 防抖节流（非重点）

- 虚拟列表

- 图片懒加载（非重点）

- web worker 线程（另一个项目使用）

- 字体优化 `font-display: swap`

- 骨架屏

- Vue框架优化

- 终极优化：SSR服务端渲染
