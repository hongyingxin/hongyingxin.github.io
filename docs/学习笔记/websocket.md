# WebSocket

## 原生 WebSocket 和 Socket.io 的对比

| 特性 | 原生 WebSocket | Socket.io |
| :--- | :--- | :--- |
| **本质** | HTML5 标准协议 | 第三方 JavaScript 库 |
| **兼容性** | 现代浏览器支持，不支持 IE9 及以下 | 极佳，支持所有浏览器 |
| **自动重连** | 不支持，需要手动写代码实现 | 内置，支持自动重连 |
| **降级策略** | 无 | 支持，可降级为 HTTP 轮询 |
| **数据收发** | 只能发送字符串或二进制 | 支持发送任何 JSON 对象，自动序列化 |
| **多房间/频道** | 无，需自行实现 | 内置（支持 Room 和 Namespaces 概念） |
| **服务端依赖** | 任何支持 WS 协议的后端 | 强依赖，前后端需要配套使用 |

## 短轮询与长轮询的区别

| 特性 | 短轮询 (Short Polling) | 长轮询 (Long Polling) |
| :--- | :--- | :--- |
| **服务器响应** | 立即响应（无论是否有数据） | 只有当有数据或超时才响应 |
| **请求频率** | 固定时间间隔，频率高 | 随数据更新频率变化，频率相对低 |
| **延迟** | 存在明显延迟（受限于轮询间隔） | 近乎实时 |
| **服务器压力** | 主要是高频的请求处理压力 | 主要是维持连接的内存和并发压力 |
| **HTTP 连接** | 请求完立即关闭 | 连接长时间挂起，返回后立即重连 |
| **实现机制** | 简单，`setInterval` 发送请求 | 稍复杂，服务器需挂起请求，客户端需递归调用 |

## 设计模式

发布订阅模式 MQTT Socket.io，很多WebSocket库都采用了这种设计。

这种事件处理机制属于广义上的发布订阅思想，但在设计模式的严谨定义中，它更准确地被称为观察者模式（Observer Pattern）。

1. 为什么它是“发布订阅”思想？

无论是 Node.js 的 wss.on('connection', ...) 还是浏览器的 socket.onmessage = ...，其核心逻辑都是：
订阅/监听：你告诉系统“如果发生了某件事（如收到消息），请执行这个函数”。
发布/触发：当系统（如 WebSocket 协议栈）检测到该事件发生时，它会主动调用你预留的函数。
这种“我先注册，你以后通知我”的异步协作方式，正是发布订阅模式的核心。

2. 严谨的区别：观察者模式 vs 发布订阅模式

虽然平时大家经常混用这两个词，但它们之间有一个关键的架构区别：是否有“中间人（Event Bus/Broker）”。

1. 观察者模式（你代码中体现的）
在你的代码中，发布者和订阅者是直接关联的：
- 服务器端：ws.on('message', ...)。这里的 ws 对象既是发布者（产生消息），也直接维护了监听者的列表。
- 客户端：this.socket.onmessage = ...。你直接在 socket 对象上挂载处理函数。
- 特点：发布者（Subject）知道订阅者（Observer）的存在。

2. 发布订阅模式（严格定义）
真正的发布订阅模式通常有一个第三方的“调度中心”或“消息中心”。
- 例子：Vue 的插件总线 bus.$on / bus.$emit，或者消息队列 Redis/RabbitMQ。
- 特点：发布者只管把消息扔给“中间人”，它完全不知道谁订阅了。订阅者也只跟“中间人”打交道。

**总结**

事件驱动编程（Event-Driven Programming），它是基于观察者模式实现的。
广义上说：它可以算作发布订阅模式的一种实现，因为它们都实现了对象间的解耦和异步通知。
专业面试/深究时：它是观察者模式，因为它没有独立的中间调度中心，发布者和订阅者之间存在直接的耦合关系。

## 什么是 WebSocket？

WebSocket 是一种在单个 TCP 连接上进行**全双工 (Full-Duplex)** 通信的协议。

### 核心特性
- **全双工**：服务器和客户端可以同时、双向地发送数据，且互不干扰。
- **长连接**：一旦建立，除非手动关闭或网络异常，连接将一直保持，避免了 HTTP 频繁握手的开销。
- **低开销**：数据包头（Frame Header）非常小（最小仅 2 字节），相比 HTTP 庞大的 Header 大大减少了带宽占用。
- **协议标识符**：
  - `ws://`：非加密通信，默认端口 80。
  - `wss://`：加密通信（TLS/SSL），默认端口 443。

### WebSocket 握手过程 (Handshake)
WebSocket 的连接始于一个标准的 HTTP 请求，通过 **Upgrade** 机制升级为 WebSocket 协议。

1. **客户端请求**：发送带特殊 Header 的 HTTP GET 请求。
   - `Upgrade: websocket`
   - `Connection: Upgrade`
   - `Sec-WebSocket-Key: [Base64 随机串]`
2. **服务器响应**：返回 `101 Switching Protocols` 状态码。
   - `Upgrade: websocket`
   - `Connection: Upgrade`
   - `Sec-WebSocket-Accept: [计算后的签名串]`

握手成功后，HTTP 通道即转变为 WebSocket 通道，后续数据均按二进制或文本帧传输。

### 浏览器端 API (原生)
```javascript
const socket = new WebSocket('ws://localhost:8080');

// 连接成功
socket.onopen = (event) => {
  socket.send('Hello Server!');
};

// 接收消息
socket.onmessage = (event) => {
  console.log('Message from server: ', event.data);
};

// 发生错误
socket.onerror = (error) => {
  console.error('WebSocket Error: ', error);
};

// 连接关闭
socket.onclose = (event) => {
  console.log('Connection closed');
};
```

1.服务端 (Node.js)
- 使用 ws 库创建一个 new WebSocket.Server({ port: 8080 })。
- 通过 wss.on('connection', ...) 监听新的客户端连接。
- 通过 ws.on('message', ...) 监听消息，并用 ws.send() 发回。

2.前端 (浏览器):
- 使用内置的 new WebSocket('ws://localhost:8080') 建立连接。
- 核心生命周期钩子有四个：
  - onopen: 连接建立。
  - onmessage: 收到数据。
  - onerror: 报错。
  - onclose: 连接断开。

### 适用场景
- 即时聊天（WhatsApp, Slack）
- 实时协作工具（Google Docs, Figma）
- 实时金融数据（股票行情、加密货币价格）
- 在线多人游戏
- 实时仪表盘/监控系统

## 核心

### 断线重连

在原生的 WebSocket 实现中，断线重连（Reconnection） 是最核心的工程化问题之一。因为原生 API 并不包含这个功能，一旦网络波动或服务器重启，连接就会彻底断开，你的应用就会失去实时性。

#### 如何设计

一个健壮的重连方案通常包含以下几个关键设计点：

A. 触发时机

不能只在 onerror 时重连，因为很多时候断开是直接触发 onclose。所以，统一在 onclose 中执行重连逻辑 是最稳妥的。

B. 延迟重连 (Delay)

不要在断开的一瞬间立即发起请求。如果服务器宕机了，成千上万个客户端同时立即重连会形成“惊群效应”，导致服务器刚启动就被冲垮。

C. 指数退避算法 (Exponential Backoff)

这是工业级的标准做法：
第 1 次失败，等待 1s。
第 2 次失败，等待 2s。
第 3 次失败，等待 4s...
设置一个最大等待上限（比如 30s）。

D. 状态保护

在重连期间，要确保不会重复创建多个 WebSocket 实例。

### 心跳检测

### 为什么需要心跳检测？

TCP自带的Keep-Alive等待时间过长，且无法检测应用层（JS线程）是否假死或卡住。

A. 假死连接 (Half-open Connection)
有时候，底层 TCP 连接已经断了（比如网线拔掉、WiFi 掉线、路由器重启），但浏览器或服务器可能还没感知到，它们依然认为 socket.readyState === OPEN。这时候你发消息，就像对着黑洞说话，没有任何反馈。

B. 空闲超时 (Idle Timeout)
很多中间件（如 Nginx, HAProxy, 运营商网关）会为了节省资源，强制关掉超过一定时间（例如 60s）没有任何数据往来的连接。心跳包可以作为“保活”手段，告诉中间件：“别杀我，我还活着。”

### 设计思路：Ping-Pong 机制

标准的做法是：定时发送，超时报错。

1. 心跳包发送：客户端每隔 10 秒发送一个{"type": "ping"}。

2. 心跳包响应：服务端收到 "ping" 后，立即回一个 {"type": "pong"}。

3. 超时判断：
  - 客户端发送 "ping" 的同时，开启一个定时器（Timeout）。
  - 如果在规定时间内收到了任何消息（或特定的 "pong"），则重置定时器。
  - 如果定时器到期还没收到消息，说明连接已死，手动执行 socket.close()，从而触发我们之前写好的 onclose 重连逻辑。

### send 方法支持json

如果支持了 JSON，通常会引入一个“信封”协议，而不是直接发业务数据。

原生 WebSocket 只有一个 onmessage 入口。如果不定义协议，你收到一个 JSON 后，很难快速判断它是“聊天记录”、“系统通知”还是“心跳回传”。

设计模式：

```js
{
  "type": "CHAT_MSG",      // 消息类型，用于分发逻辑
  "payload": { ... },      // 真正的业务数据
  "requestId": "uuid-123", // 可选：用于实现“请求-响应”模式
  "timestamp": 1678123456  // 时间戳
}
```

这样在前端 onmessage 中，你可以根据 type 直接把消息分发给不同的业务处理器（Handler）。


### 消息队列（消息必达性）

当 WebSocket 处于断线重连期间，用户可能仍在操作（比如发送聊天消息）。如果直接报错，用户体验会很差；如果直接丢弃，数据就会丢失。

消息队列（Message Queue / Buffer） 的核心作用就是：在连接断开时暂存消息，在连接恢复后自动重发。

#### 设计方案

1. 基础设计：内存缓冲区

最简单的实现方式是在 MyWebSocket 类中维护一个数组。
- 入队时机：在 send() 方法里，如果 readyState !== OPEN，不报错，而是将消息 push 进 messageQueue。
- 出队（冲刷）时机：在 onopen 事件触发时，遍历并发送队列中的所有消息。
这种设计的风险：
- 内存溢出：如果断线时间极长且发送量巨大，数组会撑爆内存。需要设置一个 maxQueueSize（比如只存最新的 100 条）。
- 乱序风险：必须严格保证 FIFO（先进先出）顺序发送。

2. 进阶设计：可靠交付 
仅仅“发出”是不够的。在不稳定的网络下，可能连接刚连上，你把队列里的消息全发了，结果 1 秒后又断了，而服务器根本没收到。

引入“确认机制 (ACK)”：
- 待确认队列 (Pending Queue)：消息发送后不立即从队列删除，而是移动到一个“等待确认”的区域。
- 消息 ID：给每一条 JSON 消息加一个唯一的 msgId (UUID 或自增 ID)。
- 服务端回执：服务器收到消息后，必须回复一个 { type: "ack", payload: { msgId: "..." } }。
- 清理逻辑：客户端收到 ack 后，才正式从队列中删除该消息。
- 重发逻辑：如果重连成功后，某些消息在一定时间内没收到 ack，则重新发送。

3. 状态同步：如何防止重复发送？
重连时最怕的就是：服务器其实收到了，但 ack 还没发出来连接就断了。 客户端重连后会再次发送同一条消息。
幂等性设计 (Idempotency)：这需要后端配合。后端记录处理过的 msgId，如果收到重复的 msgId，直接忽略业务逻辑，但依然回复 ack。

4. 流量控制

队列积压消息过多的问题

使用分批发送和优先级

5. 持久化设置

本地缓存

使用 localStorage 或 IndexedDB 存储队列。

**总结：**

一个成熟的消息队列流程：

1. 用户点击发送 -> 生成 msgId。

2.检查连接 -> 没连上？入队并存入 localStorage。

3.连接恢复 -> 触发 onopen -> 开启定时任务，每 100ms 从队列取一条发送。

4.收到 ACK -> 从队列和 localStorage 中永久移除。

5.超时未收到 ACK -> 标记为“发送失败”，重连后提醒用户或自动重试。

**队列的实现**

目前有两种方案:

一 前端维护离线队列（Array）和映射表(Map)

二 前端只维护一个消息表，通过状态字段status区分待发送、已发送、已收到

代码实现队列(FIFO)，最好使用Map。Map有一个关键特性：它会按照插入顺序进行迭代。

- 先进先出 (FIFO)：当你使用 for (let [id, msg] of myMap) 遍历时，取出的顺序一定是 set 的先后顺序。
- 高效查找 (O(1))：收到 ACK 时，你可以直接 myMap.delete(msgId)，而不需要像数组那样用 findIndex 去遍历。

**可靠通信”闭环：**
- 发送端 (前端)：
  - send -> 存入 messageMap (有序队列)并同步写入localStorage -> 真正发出。
- 网络中间层：
  - 断线 -> 消息积压在 messageMap。
  - 重连 -> flushQueue (按顺序重新发出)。
- 接收端 (后端)：
  - 收到消息 -> 查 Set 是否已处理 -> 若没处理过：存 ID、回 ACK、跑业务逻辑 -> 若处理过：只回 ACK。
- 确认闭环：
  - 前端收到 ACK -> 从 messageMap 永久移除同步删除localStorage。

### 一对多模式

WebSocket 协议（WS/WSS）本身是“点对点”的（一个客户端对应一个服务器连接），但通过在服务端维护逻辑，它可以完美实现“一对多”的广播或房间模式。

1. 房间模式的设计原理

在服务端，你需要维护一个映射关系（Mapping），通常是一个 `Map<string, Set<WebSocket>>`：

- Key：房间 ID (Room ID)。
- Value：该房间内所有活跃客户端的集合 (Set)。

**核心流程：**

- 加入房间 (Join)：客户端发送 { "type": "join", "payload": { "roomId": "room-123" } }。服务器将该 ws 对象存入 room-123 的集合。
- 离开房间 (Leave)：客户端手动离开或连接断开。服务器从集合中删除该 ws。
- 房间广播 (Broadcast)：当某人向 room-123 发消息时，服务器遍历 room-123 对应的 Set，循环调用每个 ws.send()。

2. 原生实现 vs Socket.io

**原生实现 (Native Node.js)**

你需要自己写所有逻辑：

- 手动管理 Map 和 Set。
- 手动处理断开连接时的清理（否则会造成内存泄漏）。
- 手动过滤“发送者自己”（通常广播不发给自己）。

**Socket.io (内置支持)**

Socket.io 将这个概念封装得极其简单，它原生支持 Rooms：
```js
// 加入房间
socket.join("room-123");

// 给房间里所有人发消息 (不包括自己)
socket.to("room-123").emit("message", "hello everyone");

// 给房间里所有人发消息 (包括自己)
io.in("room-123").emit("message", "hello everyone");
```

3. 分布式架构

如果你只有一台服务器，用内存里的 Map 没问题。但如果你的业务做大了，需要两台服务器做负载均衡：
- 用户 A 连在服务器 1 的 room-123。
- 用户 B 连在服务器 2 的 room-123。
- 此时，服务器 1 上的消息无法直接发给服务器 2 上的用户。

解决方案：Redis Pub/Sub (发布/订阅)
- 服务器之间不直接通信，而是都订阅 Redis 的 room-123 频道。
- 当任何一台服务器收到消息，它先发给 Redis。
- Redis 广播给所有订阅了该频道的服务器。
- 各服务器再各自发给连在自己身上的客户端。

**这就是为什么做实时通信系统，Redis 是标配的原因。**

### 鉴权设计

WebSocket 的鉴权（Authentication）比普通的 HTTP 请求要复杂一点，因为 WebSocket 是一个先握手、后长连接的过程。
目前业界主流的鉴权方案主要分为 “握手阶段鉴权” 和 “连接后第一条消息鉴权”

#### 方案一

在 HTTP 握手阶段鉴权

WebSocket 在建立连接之前，会先发送一个标准的 HTTP GET 请求（带有 Upgrade: websocket 头）。这是拦截非法连接的最佳时机，因为此时连接还没有真正升级。

A. 使用 Cookie 鉴权

原理：浏览器在发送 WebSocket 握手请求时，会自动带上同源的 Cookie。

优点：最简单，后端可以直接复用现有的 Session/Cookie 逻辑。

缺点：受跨域限制；如果是移动端（App）或非浏览器环境，处理起来较麻烦。

B. 使用 Query 参数（URL 传参）

做法：new WebSocket('ws://example.com?token=xxxx')

优点：兼容性最好，不限环境。

缺点：不安全。Token 会暴露在服务器日志、浏览器历史记录或代理服务器中。除非使用 WSS（加密传输）且 Token 是一次性的。

#### 方案二

连接建立后，发送“鉴权消息”

这是目前单页面应用（SPA）和 App 中非常流行的做法，特别是使用 JWT 的场景。

设计流程：
1. 静默连接：客户端先建立 WebSocket 连接，但不发业务数据。
2. 第一步：鉴权消息：连接建立后的 5 秒内，客户端必须发送一条 { "type": "auth", "token": "JWT_TOKEN" }。
3. 服务端校验：
  - 校验通过：回复 { "type": "auth_success" }，并将该 ws 标记为“已授权”。
  - 校验失败或超时：服务端强制断开连接。
4. 业务通信：只有标记为“已授权”的连接，服务器才处理其后续的业务消息。
这种方案的优势：完美避开了浏览器对 Header 的限制，且 JWT 放在消息体里非常安全（配合 WSS）。

#### 方案三

一次性票据

设计流程：
1. 获取票据：客户端先发一个普通的 HTTP POST 请求给后端（带上 Header 里的 Token）。
2. 下发票据：后端校验通过后，生成一个有效期极短（如 30 秒）的一次性随机字符串 ticket，存入 Redis，并返回给客户端。
3. WS 连接：客户端带着票据连接：new WebSocket('ws://example.com?ticket=TICKET_ID')。
4. 校验并删除：后端收到后去 Redis 查，查到就说明合法，然后立即删除该 ticket。

## 问题


### 心跳检测的重连计算器

**问题：**

在目前的实现中，一旦 onopen 触发，我们会立即执行 this.reconnectCount = 0。
如果一个连接很不稳定（所谓的 Flapping/抖动）：连接上了 -> 5秒后心跳超时断开 -> 自动重连。由于每次都“连上过”，计数器确实会不断重置为 0。

**方案：**

延迟重置（Stable Connection Check）：不要在 onopen 瞬间重置计数器。而是设置一个“稳定期”（比如 5-10 秒）。只有当连接持续保持 OPEN 状态超过这个时间，我们才认为这是一次“真正的成功连接”，此时再重置 reconnectCount。

**逻辑伪代码思路：**
onopen 时，开启一个 setTimeout。
如果在稳定期内连接又断了，取消这个定时器，不重置计数器，让重连间隔继续指数增加（1s -> 2s -> 4s）。
如果稳定期过了连接还在，才归零。

### 服务端回了消息但不是“pong”，算断开吗？

代码里 onmessage 第一行就是 this.resetHeartbeat()。这意味着只要服务器发了任何内容（业务消息、错误通知等），心跳超时的 setTimeout 就会被清除。

这个问题的答案取决于对心跳的定义：链路保活 vs 业务保活。


模式 A：链路保活
- 观点：只要服务器能发消息过来，不管发的是不是 pong，都说明：
TCP 连接是通的。
服务端的进程还在运行。
中间的代理（Nginx 等）不会因为超时关掉我们。
- 结论：不应该算断开。任何消息都是“生命体征”，应该重置计时器。

模式 B：严格心跳
- 观点：如果我发了 ping，你回了业务消息但没回 pong，可能说明你的“心跳处理逻辑”挂了，或者业务逻辑太忙导致心跳响应延迟。
- 结论：在这种极少数场景下，你可能只想在收到特定的 pong 时才重置计时器。

存在的问题（模式 B 的坑）：
如果采用“只有收到 pong 才重置”，会遇到一个尴尬的情况：
1. 客户端发了 ping。
2. 服务器正忙着推送 100 条业务消息，这些消息把网络带宽占满了，pong 被排在了后面。
3. 客户端因为没及时收到 pong，误杀了正在疯狂传数据的健康连接。

综合建议：
在绝大多数前端工程中，“收到任何数据都重置心跳”是最稳健的做法。心跳的主要目的是为了防止“死连接（Half-open）”，只要有任何数据包往来，连接就不是死的。

### 如何保证消息不丢失、不重复、不乱序问题

**不丢失**

例如：浏览器刷新、突然断网、服务器宕机重启（消息还在队列里）。
方案：本地缓存IndexedDB + ACK确认机制

**不重复**
例如：客户端发了消息，服务器收到了也处理了，但回传的 ACK 因为网络抖动丢了。客户端没收到 ACK，重连后会再次发送同一条消息。
方案：前端每条消息带上唯一的msgId，后端幂等性处理，使用redis维护一个已处理过的消息列表，收到重复的msgI返回ACK处理

**不乱序**
例如：虽然 WebSocket 跑在 TCP 上（TCP 保证单连接内有序），但跨连接（断线重连）时顺序会乱。比如消息 A 在旧连接发了一半断了，消息 B 在新连接先发成功了。
方案：
- 序列号，为每条消息分配一个递增的seq，后端按照seq连续性处理
- 前端发完消息后，等收到ACK后再发下一条消息

### Redis为什么最适合实时通信系统

1. Redis 提供了一个极轻量级的 发布/订阅（Pub/Sub） 模式，解决跨服务器通信的问题
2. Redis 是内存数据库，读写极快，适合高频变动、临时状态
3. Redis 提供了一个比磁盘数据库快得多的临时缓冲区 List 或 ZSet（有序集合）。
