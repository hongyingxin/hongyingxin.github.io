# 实战1

## 路由匹配

路由匹配由主应用先匹配，命中微应用容器后，再由子应用自己匹配。

1. 用户点击主应用菜单

```js
{
  title: "用户管理",
  path: "/sub-system/users",
  app: "sub-admin"
}
```

2. 主应用的vue router先处理URL，所有 /sub-system/ 路径下的都会命中SubSystemContainer.vue


```js
{
  path: "/sub-system/:page*",
  component: () => import("./views/SubSystemContainer.vue")
}
```

3. SubSystemContainer渲染<micro-app>，将子应用sub-admin挂载

```js
<micro-app
  name="sub-admin"
  url="http://localhost:4001/"
  baseroute="/sub-system"
/>
```

4. 子应用sub-admin再匹配自己的路由

```js
export default createRouter({
  history: createWebHistory(window.__MICRO_APP_BASE_ROUTE__ || "/"),
  routes
})
```

当子应用被主应用加载时，__MICRO_APP_BASE_ROUTE__就是上面的baseroute


**整个流程如下**

- 主应用路径 /sub-system/users

- 子应用 base 是 /sub-system

- 子应用实际拿到并匹配的是 /users

- 命中子应用路由

> baseroute 文档说明

```html
<micro-app name='xx' url='xx' baseroute='/my-page/'></micro-app>
```

在微前端环境下，子应用可以从window.__MICRO_APP_BASE_ROUTE__上获取baseroute的值，用于设置基础路径。
只有路由模式是native或native-scope我们才需要设置baseroute。

## 页面权限和菜单动态同步

### 场景1

主应用从后端拿到一份完整的菜单权限表。有些菜单属于子应用 A，有些属于子应用 B。

比如：A菜单有A1、A2、A3，A1是新增的模块，属于父应用，而A2、A3是旧的模块，属于子应用。这时候需要根据接口返回的路由，映射到不同的应用。

这里采用的解决方式是中心化配置。

核心思想：主应用掌握所有的菜单数据。这些数据通常由后端一个统一的权限接口返回。

实现方式：

1. 主应用登录后，请求 GET /api/menu，得到一个包含所有子系统菜单的巨大 JSON。

2. JSON 结构中包含标识，说明哪些菜单属于 sub-admin-A，哪些属于 sub-admin-B。

3. 主应用根据当前激活的子系统，动态过滤并渲染左侧菜单栏。

**优点：**权限控制非常集中，主应用能实现全局搜索菜单、全局面包屑。

**缺点：**子应用新增页面时，必须去主应用或后端配置菜单，有一定的耦合。

目前这种方案只需要在菜单新增一个字段，用于标识该菜单属于哪个子应用，其他地方不用更改。

## 多标签页的状态保持

场景：用户打开了“用户列表（子应用A）”和“订单列表（子应用B）”，在两者之间切换。切换标签时，不希望子应用重新加载。

解决：结合 micro-app 的 keep-alive 特性。主应用需要维护一个 keep-alive 列表，动态控制哪些子应用应该被缓存，哪些应该被销毁。

## 面包屑与页面标题同步

场景：主应用顶部的面包屑需要显示“首页 / 某子系统 / 某业务页”。主应用只知道自己在 /sub-app/user-detail，但不知道这个 user-detail 对应的中文名是“用户详情”。

解决：子应用在路由跳转时，通过 dispatch 向主应用发送当前页面的标题和面包屑配置，主应用接收后更新自己的 Header。

## 公共组件库的版本和主题样式

场景：主应用用 Element Plus v2.5，子应用 A 用 v2.8，子应用 B 用 Ant Design。虽然样式隔离了，但全局配置（如弹窗的 z-index、主题色）可能冲突。

解决：

主题同步：主应用切换“深色模式”时，通过全局数据通知子应用，子应用动态修改自己的 CSS 变量。

弹窗处理：子应用的弹窗默认挂载在子应用内部，如果需要全屏遮罩，需要特殊配置让弹窗挂载到主应用的 body 下。


## 跨应用组件共享

场景：子应用 A 开发了一个非常复杂的“用户选择器”组件，子应用 B 也想直接用。

解决：

低级方案：通过 iframe 嵌入。

高级方案：利用 Webpack 5 的 Module Federation (模块联邦)。主应用作为底座，子应用 A 暴露组件，子应用 B 远程引入。micro-app 完美兼容这种方案。

## 全局错误监控与日志

场景：子应用报错了，但用户没反馈，开发者不知道。

解决：在主应用统一注册 window.onerror 和 unhandledrejection。由于 micro-app 的沙箱机制，子应用的错误通常会冒泡到主应用，主应用可以统一收集并上报到 Sentry 等监控平台。
