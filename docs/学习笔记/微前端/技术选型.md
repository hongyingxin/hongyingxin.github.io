# 微前端技术选型

<!-- > 整理自：[微前端架构有哪些主流框架可以选，各有啥优劣势？](https://github.com/pro-collection/interview-question/issues/1120) -->

## 1. 选型前先看什么

微前端框架没有绝对最优，更多是围绕项目现状做取舍。选型时优先关注这几个维度：

- **技术栈兼容性**：是否需要同时接入 React、Vue、Angular 或历史项目。
- **隔离能力**：JS、CSS、路由、全局变量是否需要强隔离。
- **改造成本**：子应用是否能接受入口文件、构建配置、部署方式的改造。
- **性能要求**：是否需要模块级按需加载、依赖共享、预加载、缓存优化。
- **团队能力**：团队是否熟悉 Webpack、Vite、Web Components、iframe、single-spa 等底层机制。
- **生态成熟度**：文档、社区、生产案例、长期维护风险是否可接受。

## 2. 主流方案对比

### 2.1 qiankun

`qiankun` 是蚂蚁集团推出的微前端框架，基于 `single-spa` 封装，提供应用注册、生命周期管理、路由劫持、JS 沙箱和样式隔离等能力。

**核心特性**

- 基于 `single-spa`，支持 React、Vue、Angular 等多框架共存。
- 提供 JS 沙箱，常见方案包括 `Proxy` 沙箱和快照沙箱。
- 提供样式隔离能力，例如动态样式作用域、实验性样式隔离等。
- 有比较完整的应用生命周期和路由匹配机制。

**优势**

- 成熟稳定，有较多企业级生产实践。
- 开箱即用程度较高，适合快速搭建微前端架构。
- 生态和资料比较丰富，学习成本相对可控。

**劣势**

- 对子应用有一定侵入性，通常需要改造入口文件、导出生命周期、调整构建配置。
- 沙箱和样式隔离在复杂业务里仍可能出现边界问题。
- 大量子应用同时加载时，需要额外关注性能和资源管理。

**适用场景**

- 企业后台、中台、管理系统等大型应用。
- 技术栈混合，但希望有统一主应用、统一路由和统一接入规范的场景。
- 团队希望优先选择成熟稳定方案，而不是追求极致隔离或极致性能。

### 2.2 single-spa

`single-spa` 更像是微前端底层调度框架，主要解决子应用注册、路由匹配、加载和生命周期调度问题。

**核心特性**

- 提供应用注册、加载、卸载和路由调度能力。
- 不绑定具体前端框架。
- 可与 React、Vue、Angular 等框架配合。

**优势**

- 灵活度高，适合深度定制。
- 不强绑定具体技术栈。
- 社区较成熟，常作为其他微前端框架的底层基础。

**劣势**

- 只解决微前端调度核心，不完整解决样式隔离、JS 沙箱、应用通信等问题。
- 配置和工程化成本较高。
- 对团队架构能力要求更高。

**适用场景**

- 项目已有成熟工程体系，只需要一个微前端调度内核。
- 需要高度自定义路由、通信、隔离和资源加载策略。
- 团队愿意自己封装上层微前端能力。

### 2.3 Module Federation

`Module Federation` 是 Webpack 5 提供的模块联邦能力，本质上是跨应用的运行时模块共享机制。

**核心特性**

- 支持一个应用在运行时加载另一个应用暴露出来的模块。
- 支持依赖共享，减少公共依赖重复打包。
- 更偏向模块级集成，而不是完整应用级微前端框架。

**优势**

- 模块级复用能力强，适合组件库、业务模块、基础能力共享。
- 按需加载能力好，可以减少不必要资源加载。
- 与 Webpack 5 深度集成，适合已有 Webpack 体系的项目。

**劣势**

- 强依赖构建工具能力，Webpack 配置复杂。
- 不直接提供完整的路由、沙箱、样式隔离和通信方案。
- 多团队共享依赖时，需要严格管理版本和运行时兼容性。

**适用场景**

- 技术栈相对统一，并且使用 Webpack 5。
- 多个应用之间需要共享组件、工具库、业务模块。
- 更关注模块复用，而不是传统意义上完整子应用接入。

### 2.4 MicroApp

`MicroApp` 是京东推出的微前端框架，借鉴 Web Components 思想，通过自定义元素的方式加载子应用。

**核心特性**

- 使用类似 Web Components 的方式封装微应用。
- 提供 JS 沙箱、样式隔离、元素隔离、路由隔离等能力。
- 子应用接入成本相对较低。

**优势**

- 使用方式简单，主应用通过 `<micro-app>` 标签加载子应用。
- 对子应用侵入性相对较低。
- 隔离能力比较完整，适合快速集成历史系统。

**劣势**

- 生态规模和社区资料不如 `qiankun`。
- 某些沙箱能力依赖现代浏览器特性，例如 `Proxy`。
- 遇到复杂场景时，仍需要理解框架内部隔离机制。

**适用场景**

- 希望以较低改造成本接入已有项目。
- 技术栈混杂、历史包袱较多的中后台系统。
- 对隔离性有要求，但不想引入过重改造成本的场景。

### 2.5 Wujie

`Wujie` 是腾讯推出的微前端方案，核心思路是结合 Web Components 和 iframe，实现更强的隔离能力。

**核心特性**

- 结合 Web Components 和 iframe。
- 利用 iframe 提供天然 JS 运行环境隔离。
- 支持样式、JS、路由等多维度隔离。

**优势**

- 隔离性强，安全边界清晰。
- 适合对全局污染、样式污染、安全风险比较敏感的场景。
- 子应用运行环境相对独立。

**劣势**

- iframe 天然会带来一些体验和工程问题，例如滚动、弹窗、通信、SEO 等。
- 对现代浏览器环境更友好，老旧浏览器兼容性需要评估。
- 与普通 SPA 集成方式相比，调试和通信心智成本更高。

**适用场景**

- 金融、医疗、风控、低信任子应用等高隔离场景。
- 子应用之间需要尽量避免运行时互相影响。
- 可以接受 iframe 带来的工程复杂度。

### 2.6 Garfish

`Garfish` 是字节跳动推出的微前端框架，关注资源加载、运行时隔离和扩展能力。

**核心特性**

- 基于运行时加载和沙箱机制管理子应用。
- 支持多实例、预加载、缓存、并行加载等能力。
- 可与 Webpack、Vite 等构建工具结合。

**优势**

- 资源管理能力较强，适合复杂应用生态。
- 扩展能力较好，适合大规模多团队协作。
- 性能优化能力比较完整。

**劣势**

- 社区热度和资料相对有限。
- 上手前需要理解它的插件机制、资源调度和运行时模型。
- 对团队工程化能力有一定要求。

**适用场景**

- 大型互联网应用，子应用多、团队多、资源调度复杂。
- 对性能、缓存、并行加载有更高要求。
- 团队能接受一定学习和封装成本。

### 2.7 ICestark

`ICestark` 是阿里体系的微前端方案，强调企业级场景、状态管理和模块化能力。

**核心特性**

- 提供微应用注册、加载、通信和状态管理能力。
- 可与阿里体系工程方案结合。
- 适合复杂业务系统的模块化治理。

**优势**

- 面向企业级复杂业务场景。
- 对状态管理和跨应用通信有较完整考虑。
- 适合大型组织统一前端架构规范。

**劣势**

- 配置和理解成本偏高。
- 生态选择会受到技术体系影响。
- 对普通项目来说可能偏重。

**适用场景**

- 企业级多端或复杂业务系统。
- 需要统一状态管理、统一工程规范的组织。
- 已经在使用相关工程体系的项目。

### 2.8 Piral

`Piral` 是偏插件化架构的微前端方案，更强调应用扩展和插件市场模式。

**核心特性**

- 基于插件机制组织微应用模块。
- 支持动态加载插件。
- 提供 CLI 和插件化开发体验。

**优势**

- 适合插件化业务模式。
- 扩展性强，适合多个独立能力按需接入。
- 子模块可以独立开发和发布。

**劣势**

- 国内资料和使用案例相对少。
- 生态成熟度不如 `qiankun`、`single-spa`。
- 对复杂路由和状态管理需要额外设计。

**适用场景**

- 插件市场、低代码平台、可扩展控制台等场景。
- 中小型应用希望按插件方式组织业务能力。
- 团队能接受英文资料和较小生态。

## 3. 关键维度对比

### 3.1 技术栈兼容性

- **多框架混合**：优先考虑 `qiankun`、`single-spa`、`Garfish`。
- **低侵入接入历史项目**：优先考虑 `MicroApp`、`Wujie`。
- **统一技术栈下模块共享**：优先考虑 `Module Federation`。

### 3.2 隔离性与安全性

- **强隔离**：`Wujie`，借助 iframe 获得更明确的运行环境隔离。
- **中等隔离**：`MicroApp`、`qiankun`，通过沙箱和样式隔离降低互相影响。
- **弱隔离或需自建隔离**：`Module Federation`、`single-spa`。

如果业务对安全边界要求极高，优先考虑 `Wujie`。如果是普通后台系统，`qiankun` 或 `MicroApp` 通常已经够用。

### 3.3 性能与资源管理

- **模块级按需加载**：`Module Federation` 更有优势。
- **资源调度和缓存优化**：`Garfish` 更值得关注。
- **成熟方案下的平衡选择**：`qiankun` 更稳妥，但需要关注子应用数量和资源加载策略。

### 3.4 开发体验与学习成本

- **低学习成本**：`qiankun`、`MicroApp`。
- **中等学习成本**：`Wujie`、`Garfish`。
- **高学习成本**：`single-spa`、`Module Federation`、`ICestark`。

### 3.5 生态成熟度

- **成熟生态**：`qiankun`、`single-spa`、`Module Federation`。
- **新兴或相对小众生态**：`MicroApp`、`Wujie`、`Garfish`、`Piral`。

长期维护项目建议优先考虑成熟生态；创新项目或特定诉求明显时，可以选择更贴合场景的新兴方案。

## 4. 落地避坑

### 4.1 样式隔离

常见样式隔离方案：

- **Shadow DOM**：隔离能力强，但弹窗、下拉框、全局样式、第三方组件挂载位置可能需要特殊处理。
- **动态样式作用域**：兼容性较好，但对动态插入样式、全局选择器、样式优先级要谨慎。
- **命名规范约束**：例如 BEM、CSS Modules、命名前缀，成本低但依赖团队规范。

实践中不要只依赖框架隔离能力，业务组件库、全局样式、弹窗挂载节点也需要统一约束。

### 4.2 JS 沙箱

JS 沙箱主要解决全局变量污染、定时器、事件监听、DOM 操作等副作用问题。

需要重点关注：

- 子应用是否会修改 `window` 全局变量。
- 子应用卸载时是否清理事件监听、定时器、全局状态。
- 多实例场景下是否会互相污染。
- 第三方库是否依赖全局单例。

### 4.3 应用通信

通信方式要尽量简单，避免把微前端做成强耦合的大单体。

- 简单数据传递可以使用框架提供的 `props`、事件总线或自定义事件。
- 复杂共享状态可以考虑独立状态管理方案，但要控制共享范围。
- 跨应用通信应优先传递稳定协议，而不是直接共享内部对象。
- 避免子应用直接读写主应用内部状态。

### 4.4 路由管理

路由通常有两种模式：

- **主应用统一管理**：主应用负责子应用激活规则，适合统一入口的 SPA。
- **子应用自治**：子应用维护自己的内部路由，主应用只负责挂载和卸载。

需要注意：

- 子应用路由前缀要清晰，避免冲突。
- 刷新页面时要保证主应用和子应用都能正确命中路由。
- History 模式部署时，需要服务端 fallback 配置配合。

### 4.5 资源加载

资源加载会直接影响首屏性能和运行时稳定性。

- 关键子应用可以做预加载。
- 非关键子应用按需加载。
- 公共依赖要避免重复打包。
- 同一时间不要并发加载过多子应用。
- 子应用版本发布要注意缓存和灰度策略。

## 5. 快速决策

- **技术栈多样、追求成熟稳定**：选 `qiankun`。
- **需要底层调度能力、准备自己封装体系**：选 `single-spa`。
- **技术栈统一、核心诉求是模块共享**：选 `Module Federation`。
- **历史系统接入、希望低侵入**：选 `MicroApp`。
- **强隔离、高安全要求**：选 `Wujie`。
- **大规模应用、关注资源调度和性能**：选 `Garfish`。
- **企业级复杂业务、已有相关工程体系**：考虑 `ICestark`。
- **插件市场或插件化架构**：考虑 `Piral`。

## 6. 总结

微前端选型本质上是在 **成熟度、隔离性、性能、改造成本、团队能力** 之间做平衡。

如果没有特别极端的隔离或性能诉求，企业中后台项目通常可以优先从 `qiankun` 和 `MicroApp` 开始评估；如果项目更关注模块共享和依赖复用，可以考虑 `Module Federation`；如果子应用来源复杂、隔离要求高，则 `Wujie` 更适合。

如果想自己搭建一套微前端底座的话，推荐`single-spa`，因为它更灵活，可以自己封装上层微前端能力。小公司小团队的缘故，这套底座不太适用。

剩下在主流的`qiankun`、`micro-app`、`wujie`三选一，根据项目需求和团队能力来选择。

我需要一套侵入性低、上手成本低，这里首先`micro-app`。
