# 微前端底层原理

微前端框架表面上是在比较 `qiankun`、`MicroApp`、`Wujie`、`Garfish`、`Module Federation` 等方案，但从底层看，它们真正解决的是同几类问题：

- 子应用如何加载
- 子应用如何隔离
- 主子应用如何通信
- 路由如何调度
- 应用如何挂载和卸载

不同框架的差异，本质上是对这些问题选择了不同的实现方式。

## 1. 应用加载

微前端首先要解决的问题是：**主应用如何把另一个应用加载进来**。

常见加载方式主要有三类。

### 1.1 HTML Entry

HTML Entry 是很多应用级微前端框架常见的加载方式。

它的核心流程大致是：

1. 主应用请求子应用的 `index.html`。
2. 解析 HTML 中的 `script` 和 `link`。
3. 拉取子应用的 JS 和 CSS 资源。
4. 在主应用指定容器中执行并渲染子应用。

`qiankun`、`MicroApp`、`Garfish` 都偏向这种思路。

这种方式的特点是：加载的是一个完整应用，而不是单个模块。子应用仍然可以保留自己的工程结构、构建方式、路由和资源。

优势是适合接入已有系统，尤其是多个独立项目合并到一个主应用中。缺点是运行时要处理资源解析、样式插入、脚本执行、生命周期管理等问题。

### 1.2 模块加载

模块加载不是把完整 HTML 页面加载进来，而是在运行时加载远程模块。

`Module Federation` 就是这个方向。

它更像是：

```js
const RemoteButton = await import("remote/Button")
```

主应用关心的是远程应用暴露出来的某个组件、函数或页面模块，而不是完整子应用。

这种方式的优势是模块复用能力强，适合共享组件库、业务模块和基础能力。缺点是它不直接解决完整微前端场景中的沙箱、路由调度、样式隔离和应用卸载问题。

### 1.3 iframe 加载

iframe 是浏览器原生提供的页面嵌套能力。

它的特点是：子应用就是一个独立页面，有独立的 `window`、`document` 和执行上下文。

`Wujie` 会利用 iframe 的隔离能力，再结合 Web Components 改善传统 iframe 的使用体验。

iframe 的优势是隔离天然强，缺点是通信、弹窗、滚动、路由同步、SEO 和用户体验都需要额外处理。

## 2. JS 隔离

JS 隔离解决的是：**子应用的全局变量、事件监听、定时器、DOM 操作不能污染主应用或其他子应用**。

### 2.1 快照沙箱

快照沙箱的思路是：

1. 子应用运行前，记录当前 `window` 状态。
2. 子应用运行时，允许它修改 `window`。
3. 子应用卸载时，恢复之前的 `window` 状态。

这种方案实现相对简单，对老浏览器更友好，但它更适合单实例场景。如果多个子应用同时运行，恢复全局状态会变得复杂。

### 2.2 Proxy 沙箱

Proxy 沙箱的思路是：不让子应用直接操作真实 `window`，而是给它一个代理对象。

子应用读取或写入全局变量时，实际经过 `Proxy` 拦截：

```js
const fakeWindow = new Proxy({}, {
  get(target, key) {
    return key in target ? target[key] : window[key]
  },
  set(target, key, value) {
    target[key] = value
    return true
  }
})
```

这样子应用写入的全局变量会优先保存在自己的代理对象上，减少对真实 `window` 的污染。

`qiankun`、`MicroApp`、`Garfish` 都会使用类似思想。

Proxy 沙箱的优势是可以支持多实例，隔离能力比快照沙箱更强。缺点是依赖现代浏览器能力，并且对某些强依赖真实全局对象的第三方库，仍然可能需要特殊处理。

### 2.3 iframe 沙箱

iframe 沙箱依赖浏览器原生隔离。

每个 iframe 都有自己的全局对象：

- `iframe.contentWindow`
- `iframe.contentDocument`

这意味着子应用天然不会直接污染主应用的 `window`。

这种隔离最彻底，但通信成本也最高。主应用和子应用之间通常需要通过 `postMessage`、自定义事件或框架封装的通信机制来交换数据。

### 2.4 模块作用域

`Module Federation` 更依赖模块作用域，而不是完整 JS 沙箱。

ES Module 和打包工具会为模块提供作用域隔离，但这并不等于应用级沙箱。如果远程模块主动修改 `window`、注册全局事件或污染全局样式，仍然需要业务规范约束。

## 3. CSS 隔离

CSS 隔离解决的是：**一个子应用的样式不能影响主应用或其他子应用**。

因为 CSS 默认是全局生效的，所以微前端中的样式冲突非常常见。

### 3.1 Shadow DOM

Shadow DOM 是浏览器原生样式隔离方案。

它可以让组件拥有独立的 DOM 和样式作用域，外部样式默认不会影响内部，内部样式也不会泄漏到外部。

优点是隔离能力强。缺点是一些全局样式、弹窗组件、下拉菜单、主题变量和第三方组件库可能需要适配。

### 3.2 动态样式作用域

动态样式作用域的思路是：框架在运行时改写 CSS 选择器，给子应用样式增加作用域。

例如把：

```css
.button {
  color: red;
}
```

改成类似：

```css
micro-app[name="app1"] .button {
  color: red;
}
```

这样样式只会在当前子应用容器内生效。

这种方式兼容性较好，但不是绝对隔离。遇到 `body`、`html`、全局弹窗、动态插入样式时，仍然需要额外处理。

### 3.3 样式挂载和卸载

应用级微前端框架通常会在子应用激活时插入样式，在子应用卸载时移除样式。

这可以减少样式残留，但不能完全避免运行期间的样式冲突。

如果两个子应用同时运行，而且选择器命名冲突，仍然可能互相影响。

### 3.4 工程规范隔离

最朴素的方式是通过工程规范隔离：

- CSS Modules
- BEM
- 统一命名前缀
- 组件库样式命名规范
- 禁止随意写全局样式

这种方式不是框架能力，而是团队约束。它的成本低，但依赖执行力。

## 4. 路由调度

路由调度解决的是：**什么时候加载哪个子应用，什么时候卸载哪个子应用**。

典型流程是：

1. 监听浏览器 URL 变化。
2. 根据当前路径匹配子应用激活规则。
3. 命中规则后加载并挂载子应用。
4. 路由离开后卸载子应用。

`single-spa` 和 `qiankun` 都是典型的路由驱动型方案。

它们会关注这些浏览器 API：

- `history.pushState`
- `history.replaceState`
- `popstate`
- `hashchange`

而 `MicroApp` 更偏组件化挂载：

```html
<micro-app name="app1" url="http://localhost:3001"></micro-app>
```

这种模式下，主应用不一定只依赖全局路由来决定子应用加载，也可以像使用组件一样在某个位置挂载子应用。

`Module Federation` 通常由业务路由自己决定何时加载远程模块，框架本身不负责完整应用调度。

## 5. 应用通信

通信解决的是：**主应用和子应用之间如何交换数据**。

常见通信方式有：

- `props` 下发
- 自定义事件
- 事件总线
- 全局状态管理
- `postMessage`
- 远程模块直接调用

### 5.1 props 下发

主应用加载子应用时，把数据作为参数传下去。

这种方式简单直接，适合传递用户信息、权限信息、主题配置等相对稳定的数据。

缺点是它更适合主到子的单向传递，复杂双向通信需要额外机制。

### 5.2 事件通信

事件通信适合低耦合场景。

主应用和子应用只约定事件名称和数据格式，不直接依赖彼此内部实现。

这种方式的关键是事件协议要稳定，否则应用多了之后会变得难以维护。

### 5.3 全局状态

多个应用共享同一个 store，看起来很方便，但也容易破坏微前端边界。

如果大量业务状态都放进全局 store，子应用之间会重新耦合成一个大单体。

更好的方式是只共享真正全局的数据，例如登录态、主题、语言、权限，不要共享具体页面的内部业务状态。

### 5.4 postMessage

iframe 场景常用 `postMessage`。

它的优势是浏览器原生支持跨窗口通信。缺点是使用时要注意来源校验、消息格式约定和异步通信复杂度。

### 5.5 远程模块调用

`Module Federation` 场景下，远程模块可以像普通模块一样被调用。

这种体验很好，但也最容易让边界变模糊。因为调用太方便，团队可能会把跨应用依赖写得非常深。

## 6. 生命周期

应用级微前端通常会抽象出统一生命周期。

常见生命周期包括：

- `bootstrap`：应用初始化。
- `mount`：应用挂载到页面。
- `unmount`：应用从页面卸载。
- `update`：应用参数更新。

生命周期的意义是让主应用可以统一管理子应用的启动、渲染、销毁和更新。

比如路由命中时调用 `mount`，路由离开时调用 `unmount`。如果子应用没有在卸载阶段清理副作用，就可能出现事件监听残留、定时器残留、样式残留等问题。

`Module Federation` 没有统一的应用生命周期。它加载的是模块，模块后续的生命周期通常交给 React、Vue 等框架自己管理。

## 7. 框架差异总结

从底层看，几个常见方案可以这样理解：

- `single-spa`：核心是路由调度和应用生命周期，不负责完整沙箱和样式隔离。
- `qiankun`：在 `single-spa` 之上补齐 HTML Entry、JS 沙箱、样式隔离和应用通信。
- `MicroApp`：借鉴 Web Components 思想，用自定义元素方式挂载子应用，强调低侵入接入。
- `Wujie`：结合 Web Components 和 iframe，利用浏览器原生能力提供更强隔离。
- `Garfish`：应用级加载、沙箱和插件化资源调度能力更突出。
- `Module Federation`：运行时模块共享，适合模块级复用，不是传统完整应用隔离方案。

## 8. 总结

微前端的底层问题可以归纳为四个字：**加载、隔离、通信、卸载**。

不同框架之间的差异，不是单纯谁更强，而是它们在这四件事上的取舍不同：

- HTML Entry 适合完整应用接入。
- Module Federation 适合远程模块共享。
- Proxy 沙箱适合模拟隔离环境。
- iframe 沙箱适合强隔离场景。
- Shadow DOM 和样式作用域解决 CSS 冲突。
- 生命周期管理决定子应用能否干净挂载和卸载。

所以做微前端选型时，不应该只看框架热度，而是要先判断项目到底需要哪种底层模型。
