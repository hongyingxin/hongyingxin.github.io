## 4月14

在面试过程中，面试官提到一个有趣的问题：你们如何去刷新CDN缓存的？

结合最近工作中遇到的问题：刷新CDN后用户还是访问旧的页面。

对缓存有了进一步的认知。

首先需要区别**HTTP缓存**和**CDN缓存**。

**HTTP缓存：** 发生在用户自己的电脑浏览器上，这是最“近”的缓存。如果命中，浏览器直接从硬盘读数据，这跟刷新CDN没任何关联，因为请求根本没发出去

**CDN缓存：** 发生在CDN节点服务器上，这是中间站。当用户自己的电脑浏览器没有缓存发起请求时，CDN会拦截这个请求。这时如果CDN有，它就给你；如果没有，才去源站服务器要。

| 维度 | HTTP 缓存 (浏览器) | CDN 缓存 |
| :--- | :--- | :--- |
| 存储位置 | 用户本地磁盘/内存 | CDN 运营商的边缘服务器 |
| 生效前提 | 浏览器发现 URL 没变，且没过期 | 浏览器发出了请求，但没到达源站 |
| 控制方式 | Cache-Control, Expires | CDN 后台配置 或 Cache-Control |
| 手动刷新的影响 | 无效（你管不到用户电脑） | 有效（清除了云端的备份） |
| 解决办法 | 改变 URL (加 ?t=1 或文件名哈希) | 在 CDN 后台点“刷新 URL” |

### 第一个问题：为什么刷新了CDN缓存用户还是访问旧页面

HTTP和CDN共用了同一个控制指令：`Cache-Control`。

刷新CDN缓存，实际上是清除了CDN节点的缓存，但是浏览器仍然会使用本地缓存。我们无法远程删除用户浏览器的那份缓存。

### 第二个问题：通过URL后面拼接`t=时间戳`这种带参数的方式能刷新

这种本质上是利用了“缓存击穿”的原理，同时击穿了两层缓存（浏览器和CDN）。之所以能生效，是因为浏览器和CDN衡量一个资源是否被缓存过去，是基于完整URL作为唯一标识符的。

- `index.html`

- `index.htm?t=1`

- `index.html?t=2`

这三者虽然指向物理磁盘上的同一个文件，但对浏览器来讲，这时三个完全不同的请求。浏览器在缓存库里找不到这个新地址，就会被迫跳过本地强缓存，去CDN重新请求。

CDN也是同理，会重新请求源站，返回新的内容。

另外`index.html?t=1#/routeName`Hash符号后面的内容是不会发送到服务器上的，当访问`index.html?t=1#/routeName`时，请求的真正主体是`index.html?t=1`。

---

因为在业务上都是webView嵌套的H5页面，所以我们能使用手动清除缓存的方式。这里也需要进一步区分，手机系统层面和APP层面。

### 手机系统层面

- Android（系统设置 -> 应用管理 -> 清除缓存）

这种是从物理层面删除APP目录下所有的`cache`文件夹，包括`WebView`的内核缓存（磁盘缓存、Service Worker、LocalStorage等）。

- IOS（系统设置 -> 通用 -> IPhone存储空间 -> 卸载/删除）

IOS基本是无效的，只能靠删除APP重新安装来实现缓存清除。

### APP层面

这取决了开发人员的代码实现，有没有调用 Webview 的清理 API（例如 Android 的 Webview.clearCache(true) 或 iOS 的 WKWebsiteDataStore）

---

绕远了，回到问题一本身。为什么刷新CDN后用户还是访问旧页面？

通过`index.html`文件的`cache-control: max-age=86400`，表示该文件在CDN缓存中有效期为86400秒（1天）。这也就是我们刷新CDN后用户还是访问旧页面的根本原因。

正常情况下，CDN缓存需要区别文件设置不同的缓存策略。`index.html`文件一般设置为不缓存`Cache-Control: no-cache`或者缓存时间较短`Cache-Control: max-age=30`。其它JS/CSS/图片等资源文件设置为缓存时间较长`Cache-Control: max-age=86400`。

另外由于浏览器和CDN公用一个控制指令：`Cache-Control`，可以使用HTTP协议专门设计的一个指令来区分“公共缓存（CDN）”和“私有缓存(浏览器)”。

`max-age`是公共缓存，告诉浏览器和 CDN 都要缓存多久。

`s-maxage`是私有缓存，只告诉 CDN 缓存多久，浏览器不缓存。

> 上面两个API未经过测试，仅供参考。

## 4月15

### axios如何取消请求

这也算比较经典的面试题了。

在Axios中，取消请求主要依赖于Web标准的`AbortController`。在此之前，Axios使用的是自带的`CancelToken`。

**使用方法**

```js
// 1. 创建控制器实例
const controller = new AbortController();

axios.get('/api/data', {
   // 2. 将信号传递给请求
   signal: controller.signal
}).catch(err => {
   // 3. 处理取消后的逻辑
   if (axios.isCancel(err)) {
       console.log('请求已被手动取消:', err.message);
   } else {
       console.log('其他错误:', err);
   }
});

// 4. 在需要的地方触发取消
controller.abort();
```

---

由上面的取消请求，又引申出另一个问题：如何自动取消重复请求

比如在搜索框联想、页面切换，通常希望当新请求发起时，自动取消之前的同类请求。

```js
let controller;

function fetchData() {
  // 如果之前有请求在进行，直接取消它
  if (controller) {
    controller.abort();
  }

  controller = new AbortController();

  axios.get('/api/search', { signal: controller.signal })
    .then(res => console.log(res.data))
    .catch(err => {
      if (axios.isCancel(err)) return;
      // 处理真实异常
    });
}
```

### Promise无法取消Pending

Promise的本质是一个简单的状态机，拥有三种状态：Pending、Fulfilled、Rejected。

Promise标准（A规范）只定义了状态如何从Pending到Fulfilled或Rejected。它没有定义第四种状态（Cancelled）

### 比较独特的面试题

群友吹水提到一道独特的面试题：例如`let a = await requestA() requestB({a.userId})`这种场景，如何去掉await，还能保证先A后B，这种时候promise解决不了。

其实一开始我也理解不了，在异步编程中，去掉`await`但仍需要保证执行顺序。这明面上是根本不可能实现的。

但换个角度，从系统架构设计的层面解耦上看，确实存在这种需求。即响应式或流式的任务。

- 首先是响应式流派，使用RxJS这种观察者的库，可以将异步请求视为“事件流”。通过操作符来声明逻辑关系，而不是手动控制等待。即管道操作符。

```js
import { from } from 'rxjs';
import { switchMap } from 'rxjs/operators';

// 声明一个流：当 A 完成时，自动将结果映射到 B
const stream$ = from(requestA()).pipe(
  switchMap(user => requestB(user.userId))
);

// 只有在需要结果时才订阅
stream$.subscribe(result => {
  console.log("B 的结果:", result);
});
```

- 消息总线/事件驱动，这是解决“系统架构”耦合的经典手段。A执行完后，它不直接调用B，而是发布一个消息。

```js
// A 模块只管发
const a = await requestA(); 
eventBus.emit('A_DONE', a.userId);

// B 模块独立运行
eventBus.on('A_DONE', (id) => {
  requestB(id);
});
```

- 模仿`Koa`或`Redux`的中间件机制，将任务链条化。核心是任务调度器，它负责按照`[TaskA、TaskB]`的顺序执行，并自动将上一个任务的输出作为下一个任务的输入。可以理解成类似于React的Fiber架构，即调度器或协调器。
